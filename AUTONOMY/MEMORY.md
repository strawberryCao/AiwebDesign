## Durable operating lessons
- Product changes must be owned by one closed-loop run; stateless multi-stage relays create races and context loss.
- Use a fixed automation kernel plus mutable repository objective. Do not rewrite the entire automation prompt every cycle.
- Deploy and inspect the untouched baseline before editing.
- Separate builder self-evaluation from independent critic evaluation.
- Keep short-term hypotheses separate from long-term memory.
- Evidence must include commit identity, deployment identity and observable verification; unsupported claims are not memory.

## Hard constraints
- No copied third-party code, text, logos, models, textures, video, audio or proprietary assets.
- No source edit while another valid primary lease exists.
- No push after failed syntax check or production build.
- No completion claim until two consecutive independently verified scores reach at least 92/100.

## Promotion rule
Add a new durable lesson only when:
1. the user explicitly establishes it; or
2. two independent verified runs support it; or
3. an authoritative technical source plus one verified local experiment support it.

When a durable lesson becomes false, do not silently delete it. Record the reversal in `DECISIONS.md`, then revise this file.
